## 1. 用途

这份文档补充 README 首页没有展开的三类内容：

1. 奖励模型训练时，prompt 是怎么构造的
2. `11-30` 和 `31-60` 的真值为什么适合用奖励模型救援
3. 为什么当前冻结版本把 `61-100` 先定义成保守的名次提升器

## 2. Prompt 构造样本

当前 Stage11 的训练样本不是“把答案直接告诉模型”，而是给模型一个局部候选组，让它只根据当前可见信息判断：

- 用户反复喜欢什么
- 用户明确回避什么
- 近期意图是什么
- 当前候选和附近竞争者分别有什么语义匹配和冲突
- 当前候选位于哪个排名窗口

`31-60` 训练样本里的真实 prompt 结构大致如下：

```text
You are a recommendation ranking judge.
Evaluate one focus candidate inside a small local rescue slate for the same user.
Current rank is context only and is not by itself the answer.

User:
- user_focus
- recent_intent
- user_avoid
- history_pattern
- user_evidence

Local ranking context:
- Focus candidate is in the mid rescue band (31-60)
- Local slate contains 4 rivals
- same_band=1, boundary=2, head_anchor=1, deep=0

Focus candidate:
- business_profile
- user_match_points
- user_conflict_points

Rivals 1-4:
- same-slice blocker / boundary blocker / head anchor
```

这个 prompt 的关键点是：

1. 它使用的是当前候选所在窗口、局部竞争者、用户侧可见证据和商户侧语义证据。
2. 它不会把“这个候选最终是不是正样本”直接告诉模型。
3. 它也不会按真值分段去做推理路由；真值只在离线训练和评估时作为监督信号使用。

所以当前 Stage11 的增益，来自“在局部竞争关系里识别被低估的候选”，不是来自标签泄露。

## 3. 案例一：`11-30` 真值为什么能被救

这里看一个真实用户：

- `user_idx = 1072`
- 真值候选 `item_idx = 58`
- `learned_rank = 17`
- `blend_rank = 4`
- `final_rank = 8`
- `route_band = boundary_11_30`
- `reward_score = 13.4375`
- `sidecar_norm = 0.9474`
- `rescue_bonus = 0.7579`

这类样本的特点是：

- 结构化精排已经把真值送到了边界附近，但前面仍然有一批看起来很稳的边界候选和 `31-40` 候选。
- 真值本身不是“基线很差的深段样本”，而是“已经接近前沿，但还缺最后一脚”的边界样本。
- 奖励模型对这个真值给出了很高的局部判断分，说明它在用户偏好、近期意图和局部竞争关系里都更站得住。

为什么这种样本适合奖励模型来救：

1. 结构化精排已经给出了稳定骨架，不需要大模型重排全榜。
2. 边界样本的成败，往往取决于“几个局部对手之间谁更适合这个用户”，这正是奖励模型擅长的局部比较问题。
3. 这个判断过程只依赖当前候选窗口、用户语义和竞争者语义，不需要知道真值答案。

所以 `11-30` 专家的作用，不是推翻结构化精排，而是在边界附近做更细粒度的取舍。

## 4. 案例二：`31-60` 真值为什么能被救

再看一个中段真值被救回前十的真实用户：

- `user_idx = 1940`
- 真值候选 `item_idx = 92`
- `learned_rank = 36`
- `blend_rank = 1`
- `final_rank = 2`
- `route_band = rescue_31_40`
- `reward_score = 11.3125`
- `sidecar_norm = 1.0000`
- `rescue_bonus = 1.0350`

这个样本的 top10 里仍然有大量 `11-30` 候选，但真值最后能冲到第 2，原因不是简单“把中段样本硬推上来”，而是：

1. 这个真值在局部候选组里，和用户近期兴趣、历史偏好、竞争者差异的匹配关系明显更强。
2. `31-60` 专家并不是拿它去和全榜所有候选比，而是让它在自己最相关的一批近邻竞争者里先证明自己。
3. 当局部比较已经非常明确时，再通过受控的融合和门槛规则，把这种“局部明显更优”的候选释放到前排。

为什么这种样本更适合奖励模型而不是继续堆结构化特征：

- 结构化精排擅长搭建全局骨架，但中段样本往往输在“局部语义差一点没被看出来”。
- 奖励模型更容易在同一用户的小候选组里识别“哪个中段样本其实比前面的几个边界样本更合适”。
- 这个过程仍然只用当前候选窗口和局部证据，不依赖真值位置。

所以 `31-60` 专家的价值，主要体现为中段救援，而不是全榜重排。

## 5. 案例三：`61-100` 为什么先只做保守名次提升

当前冻结版本里，`61-100` 专家已经训练成功，但策略上保持保守：

- 先做深段排序修正
- 不强推前排
- 不为了让 deep 样本进前二十或前三十而破坏当前主链稳定性

这样处理的原因是：

- 深段候选离前排太远，直接硬推风险高
- 当前版本更看重整条链可解释、可回退、可冻结

因此，当前版本把 `61-100` 的价值定义成：

- 提升深段样本的排序位置
- 为后续更激进的 deep policy 留出空间

## 6. 当前结论

当前 Stage11 的有效性可以分成三部分理解：

1. `11-30` 专家负责边界附近的细粒度判断，适合做前十边界救援。
2. `31-60` 专家负责中段样本的局部竞争判断，适合把真正被低估的样本释放到前排。
3. `61-100` 专家已经学到有效信号，但当前冻结版本只把它作为保守的名次提升器使用。
