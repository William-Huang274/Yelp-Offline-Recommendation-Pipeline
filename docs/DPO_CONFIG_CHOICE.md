# DPO 配置选择指南
# 基于实际数据分析结果

## 数据分析结果

你的 prompt 实际长度：
- 平均: 432 tokens
- P95: 477 tokens (95%的数据 ≤ 477)
- P99: 525 tokens (99%的数据 ≤ 525)
- 最大: 687 tokens

## 配置方案对比

### 方案 1: 保守配置 (推荐) ✓
**文件**: `config/dpo_safe_512.env`

```
MAX_SEQ_LEN=512
LoRA_R=8
数据覆盖率: 98.7%
截断样本: 460个 (1.3%)
显存需求: ~7.5GB
```

**优点**:
- 显存安全，不会 OOM
- 覆盖 98.7% 数据，截断很少
- 训练稳定

**缺点**:
- 460个样本会被截断（平均截断 43 tokens）

**适合**:
- 首次训练
- 显存不确定
- 追求稳定性

---

### 方案 2: 优化配置 (激进)
**文件**: `config/dpo_optimized_640.env`

```
MAX_SEQ_LEN=640
LoRA_R=6 (降低以补偿)
数据覆盖率: 99.9%
截断样本: 28个 (0.1%)
显存需求: ~8.5-9GB
```

**优点**:
- 几乎不截断数据（只有 28 个样本）
- 保留更完整的信息

**缺点**:
- 显存接近上限，可能 OOM
- LoRA rank 降低到 6，可能影响效果
- 训练速度更慢

**适合**:
- 显存充足时
- 追求最佳数据覆盖
- 愿意承担 OOM 风险

---

## 我的建议

### 推荐：先用方案 1 (512)

**理由**:
1. **截断影响不大**: 只有 1.3% 数据被截断，平均只截掉 43 tokens
2. **显存安全**: 7.5GB 留有余地，不会 OOM
3. **效果可能差不多**: 被截断的通常是末尾的次要信息

### 如果方案 1 效果不好，再试方案 2 (640)

**步骤**:
1. 先用 512 训练一轮
2. 评估效果
3. 如果觉得截断影响大，再用 640 重新训练

## 截断的影响分析

被截断的 460 个样本（1.3%）：
- 平均被截断 43 tokens
- 通常是 prompt 末尾的排序特征或次要信息
- 核心的用户偏好和候选物品信息通常在前面，不会被截断

**结论**: 1.3% 的截断率对整体效果影响很小。

---

## 如果遇到 OOM

方案 2 (640) 如果 OOM，可以：
1. 降低 LoRA_R 到 4
2. 减少 DPO_MAX_PAIRS 到 2
3. 回退到方案 1 (512)
