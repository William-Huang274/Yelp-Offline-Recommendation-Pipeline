# 设计取舍

[English](./design_choices.md) | [中文](./design_choices.zh-CN.md)

这份文档集中说明从根 README 挪出的核心设计选择。

## 为什么用 bounded LLM rerank，而不是 full-list rerank？

这个项目没有让 LLM 或奖励模型对全量候选做重排。

当前方案只对 `Stage10` 输出窗口做重排，原因是：

- 成本更低
- 行为更可控
- 更容易回退
- 对前排结果的扰动更小

因此，`Stage10` 仍然是全局排序骨架，`Stage11` 只是一个有边界的增益层。

## 为什么先验证 `bucket10`，再扩到 `bucket5`？

这条实验路线是刻意设计的。

- `bucket10` 是高交互用户集，用户语义更丰富，行为信号更干净。
- 它最适合先验证召回、精排和救援式重排三层设计是否作为一套系统成立。
- 方向确认之后，再扩到 `bucket5`，用于验证更大覆盖面、更接近主展示口径的效果。
- `bucket2` 保留用来检验 `Stage09 -> Stage10` 在更冷用户集上的可迁移性。

## 怎么控制数据泄露？

当前 Stage11 线按“不在推理时暴露标签”的原则设计：

- 专家路由依据候选当前所在的排名窗口，而不是隐藏真值分段
- shortlist reranking 只使用当前分数和当前名次
- 真值只参与训练监督和离线评估

## 当前三层职责

### 召回路由（`Stage09`）

这一层把不同来源的候选组织成统一候选池，并控制召回预算、候选通道和挑战者通道。

### 结构化精排（`Stage10`）

这一层是全局排序骨架。它消费匹配特征、文本特征、相对交叉特征和分组差距特征，
输出主排序结果。

### 救援式重排（`Stage11`）

这一层只作用在 `Stage10` 输出的有边界候选窗口上。当前方案包括：

- 分段专家模型
- 候选短名单重排
- 有边界的门槛与保护规则
